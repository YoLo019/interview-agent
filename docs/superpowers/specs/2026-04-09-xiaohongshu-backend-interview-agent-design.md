# 小红书后端 / AI 应用开发面经日报 Agent 设计

## 1. 背景与目标

目标是实现一个每天自动运行的 agent，从小红书抓取前一天发布的后端相关面经与 AI 应用开发 / Agent 工程化相关面经，完成筛选、去重、知识点归类、重点题目提炼和标准答案生成，最终输出一篇可直接阅读的飞书日报。

该 agent 的核心价值不是保存原始帖子，而是每天为用户提供两类复习信息：

- 最新值得看的面经
- 当天最值得复习的高频知识点和高频题目

## 2. 用户确认的关键约束

- 抓取方式使用登录态浏览器自动化。
- 输出端只面向飞书日报，不要求飞书多维表格或腾讯文档作为用户可见交付物。
- 报告主题聚焦于两类内容：
  - 后端相关面经
  - AI 应用开发 / Agent 工程化相关面经
- Go / Golang / C++ 相关内容明确排除。
- 帖子即使未明确写出后端岗位，只要题目语义明显属于后端或 AI / Agent 工程化，也应纳入。
- 评论区内容不参与抓取与分析。
- 默认先分析标题和正文；只有当标题和正文不足以提取有效题目或相关信息时，才对图片执行 OCR。
- 题目抽取仅支持换行列表式结构，包括有编号和无编号的一行一题，不支持连续描述式正文。
- 日报不展示岗位、难度、链接。
- 日报中的重点题目需要附带标准面试版回答，而不仅是题目列表。
- 处理口径按自然日批处理。
  - 例如 2026-04-09 凌晨运行时，处理 2026-04-08 00:00-23:59 发布的帖子。
  - 用户在 2026-04-09 早上查看 2026-04-08 的日报。

## 3. 非目标

以下内容不属于首版目标：

- 实时分钟级监控与推送
- 泛技术岗面经聚合
- 依赖评论区补充信息进行主分析
- 为所有题目生成长篇教程式讲解
- 面向用户展示原始帖子链接、岗位字段、难度字段
- 重 Agent 自主规划架构

## 4. 推荐方案

采用混合式方案：

- 浏览器自动化负责抓取
- 规则负责初筛和硬排除
- LLM 负责语义判定、信息抽取、题目标准化、知识点标签和题目作答

选择该方案的原因：

- 小红书帖子结构松散，单靠规则漏判和误判都会较多。
- 完全交给多 Agent 自主决策，首版稳定性和可调试性不足。
- 混合式方案能在稳定性、可控性和理解能力之间取得平衡。

## 5. 日报输出设计

飞书日报固定输出以下六个区块：

### 5.1 今日概览

包含：

- 目标日期候选帖子数
- 筛出的有效面经数
- 去重后的有效面经数
- 后端题与 AI / Agent 题的大致占比

### 5.2 最新面经

面向当天新增的高价值面经，按信息质量和复习价值排序展示。

每条仅包含：

- 公司名
- 面试轮次
- 核心题目

不展示：

- 具体发布时间
- 岗位
- 难度
- 链接
- 帖子摘要

### 5.3 高频知识点

列出当天高频出现的知识点，并允许分为两个子类观察：

- 后端基础类，例如 MySQL、Redis、JVM、缓存、消息队列、分布式、Spring
- AI 应用开发 / Agent 类，例如 RAG、向量库、Prompt 编排、Function Calling、Tool Use、Memory、评测与观测

### 5.4 高频题目与标准回答

这是日报核心区块。

只针对当天重点题目和趋势上升题目生成标准面试版回答，每题输出：

- 题目
- 标准回答
- 考察点
- 回答结构
- 常见追问

回答风格要求：

- 可直接用于面试准备
- 不是一句话速记
- 也不是教程式长文

### 5.5 趋势变化

对比近 7 天历史数据，指出当天明显上升的话题，包括但不限于：

- 升温知识点
- 升温题型
- AI / Agent 相关题占比变化

### 5.6 重点观察

用极少量高信号结论提醒用户，例如：

- 某公司当天出现多篇相似面经
- 某类题连续多天高频
- 某个 AI / Agent 主题开始从概念题转向工程落地题

## 6. 数据处理流程

系统按批处理方式执行以下流水线：

1. 抓取帖子
2. 提取标题和正文
3. 判断文本信息是否已足够
4. 仅在文本不足时对图片执行 OCR
5. 进行相关性判定
6. 按换行列表结构抽取题目
7. 标准化题目表达
8. 生成知识点标签
9. 对帖子和题目做去重归并
10. 统计高频知识点与高频题目
11. 对重点题目生成标准答案
12. 生成飞书日报

## 7. 抓取与候选内容获取

### 7.1 时间窗口

每次任务只处理目标自然日内发布的帖子，不处理跨天实时更新。

### 7.2 输入内容与 OCR 决策链

为解决 OCR "先有鸡还是先有蛋" 的问题，采用**两级决策**而非简单的"文本不足→OCR"：

**第一级：低成本文本信号判定**

每条帖子先用标题和正文（如果有）进行两步低成本检查：

1. 硬排除检查：标题和正文中出现 Go/Golang/C++/前端/产品/运营 → 直接跳过，不做 OCR
2. 候选信号检查：标题和正文中的关键词是否命中召回池（第 8.1 节）或包含面试信号词（"面经"、"面"、"面试"、"一面"、"二面"、"终面"、"手撕"、"八股"）→ 通过则进入第二级

**第二级：OCR 的分层触发**

| 文本状态 | 行为 |
|----------|------|
| 文本中已提取到足够的换行列表式题目 | 不触发 OCR |
| 文本通过了第一级检查，但题目不足（含 0 题）| 触发 OCR |
| 文本未通过第一级检查 | 不触发 OCR，直接丢弃 |

**输入优先级**：

每条帖子最终使用的分析内容按以下优先级合并：

1. 标题
2. 正文
3. 图片 OCR 文本（仅在触发 OCR 时补充，追加到正文末尾）

评论区内容明确不参与分析。

### 7.3 搜索与发现策略

每天按以下多路搜索策略从小红书发现候选帖子，每路独立执行、结果合并去重：

**搜索词组合**：

终端用户无法直接按时间筛选小红书内容，因此搜索词本身需要嵌入时间信号。每次任务使用以下两类搜索词：

A 类：固定面经搜索词（用于召回通用后端面经和 AI/Agent 面经）

```
后端面经
后端面试
服务端面经
Java 面经
AI 应用开发 面经
Agent 面经
LLM 面经
RAG 面经
```

B 类：锚定搜索词（用于发现可能不写"面经"两个字但题目内容属于目标领域的帖子）

每个运行日从以下池中随机抽取 3 个词各搜一次，避免固定搜索模式触发反爬：

```
Redis 面试
MySQL 面试
分布式 面试
JVM 面试
MQ 面试
微服务 面试
MCP 面试
Function Calling 面试
Prompt 面试
向量数据库 面试
```

**搜索结果处理**：

1. 每个搜索词打开小红书搜索结果页（按"最新"排序）
2. 只收集搜索结果卡片中显示的帖子 URL，不打开每篇帖子的详情页
3. 收集到 URL 池后统一去重
4. 再去逐一打开每个帖子详情页，提取标题和正文

**时间过滤**：

- 搜索阶段不做时间过滤（搜索结果页已按最新排序，且小红书的搜索页没有精确时间筛选控件）
- 在帖子详情页提取发布时间后，再按目标日期窗口（前一天 00:00-23:59）过滤

**分页策略**：

- 每个搜索词最多向下滚动 5 次（约 50-80 个结果）
- 搜索结果池上限 200 个候选 URL
- 详情页抓取上限 100 篇（超出则随机采样 100 篇）

题目抽取只处理换行列表式内容：

- 有编号的一行一题
- 无编号的一行一题

不处理连续描述式正文。

## 8. 召回与相关性判定

### 8.1 候选召回

候选召回分为两类关键词池：

后端关键词池示例：

- Java
- 服务端
- Redis
- MySQL
- JVM
- MQ
- 消息队列
- 缓存
- 分布式
- Spring
- 微服务

AI 应用开发 / Agent 关键词池示例：

- LLM
- RAG
- Agent
- MCP
- Function Calling
- Prompt
- 向量库
- Embedding
- Tool Calling
- Memory
- 工作流
- 评测
- 观测

原则是先尽量放宽召回，再进入二次判定。

### 8.2 硬排除规则

出现以下明确排除信号时，优先判为无效：

- Go
- Golang
- C++
- 前端
- 产品
- 运营

### 8.3 语义判定

使用规则加 LLM 的混合判定，将帖子分为三类：

- 有效后端面经
- 有效 AI 应用开发 / Agent 面经
- 无效或低相关内容

即使帖子没有写后端岗位，只要题目语义明显属于后端或 AI / Agent 工程化，也允许纳入。

## 9. 信息抽取与内容提炼

对被判定为有效且题目结构符合要求的帖子，保留以下结构化结果：

- 公司名
- 面试轮次
- 原始题目列表
- 题目标准化表达
- 知识点标签

规则与 LLM 的职责划分如下：

- 规则抽取显式信息，例如公司名、轮次、明确出现的技术词
- 规则按换行列表结构切分题目，支持有编号和无编号的一行一题
- LLM 负责对已切分题目做标准化归并，并从原始内容中归纳知识点标签

约束：

- 不生成帖子摘要
- 不补造原帖没有出现的事实
- 不对连续描述式正文做题目切分
- 如果公司名、轮次或题目不明确，则保留为空或标记不明确

## 10. 去重与归并

### 10.1 帖子去重

目标是避免同一帖子重复抓取、改标题重发或高度相似内容重复进入候选集。

### 10.2 题目去重

不同帖子中的同义问题要归并为同一标准题目，例如：

- Redis 为什么快
- 为什么 Redis 这么快

需要归到同一个题目簇中，用于后续高频统计。

### 10.3 去重目标

系统真正希望保留的是：

- 当天新增且值得关注的面经
- 当天真正高频的题目与知识点

而不是保留尽可能多的原始帖子副本。

## 11. 排序与统计逻辑

### 11.1 最新面经排序

由于日报按自然日输出，不再按具体发布时间排序。

最新面经按以下维度综合排序：

- 信息质量
- 复习价值

### 11.2 高频知识点排序

按当天出现频次排序，并与近 7 天历史基线做对比，识别升温主题。

### 11.3 高频题目排序

按以下因素综合排序：

- 当天重复出现次数
- 是否属于核心后端基础题
- 是否属于近期升温的 AI / Agent 工程化题

## 12. 题目答案生成策略

答案生成只覆盖日报中的重点题目和趋势上升题目，不覆盖所有抽取到的问题。

每题输出格式固定为：

- 题目
- 标准回答
- 考察点
- 回答结构
- 常见追问

答案质量要求：

- 风格为标准面试版回答
- 强调可直接复习和复述
- 避免泛泛空话

降级策略：

- 如果题目语义不完整或上下文明显不足，不强行生成答案
- 可保留原始题目或直接不纳入重点区

## 13. 后台存储要求

虽然用户只阅读飞书日报，但系统内部仍需要保留轻量历史存储，以支持：

- 跨天去重
- 近 7 天趋势比较
- 高频题目统计
- 重试与失败恢复

该存储不需要作为用户可见产品，但对日报质量是必要条件。

## 14. 调度与运行方式

建议调度方式：

- 每日凌晨执行抓取与处理
- 每日早上输出飞书日报

首版仅保留一天一次的主任务，不做白天多次增量更新。

日报标题建议使用目标日期，例如：

- 4月8日 小红书后端 / AI 应用开发面经日报

## 15. 失败处理

### 15.1 抓取失败

当天不发送正式日报，只发送失败通知，说明目标日期数据抓取失败。

### 15.2 部分失败

如果抓取成功，但 OCR、分类或答案生成部分失败，则日报照发，但在顶部标记样本不足或部分内容降级生成。

### 15.3 飞书写入失败

先将生成结果保存在本地并自动重试；若仍失败，再发告警。

## 16. 质量控制

为减少噪音和误判，至少设置以下三道质量门槛：

- 相关性门槛
- 题目完整度门槛
- 作答可信度门槛

规则如下：

- 非后端、非 AI 应用开发 / Agent 工程化内容不进入日报重点区
- 只有模糊描述、无法提炼出问题的帖子不进入重点题目区
- 不符合换行列表式结构的帖子不进入重点题目区
- 无法稳定生成答案的问题不强行回答

## 17. 验证策略

首版上线前，应使用一批历史帖子做人工抽查，重点验证：

- 非目标内容是否被错误纳入
- Go / Golang / C++ 是否排除干净
- 题目标准化是否合理
- 高频知识点是否真实反映当日趋势
- 题目答案是否接近真实可用的面试回答

## 18. 后续规划入口

本设计文档只定义产品与系统设计，不展开具体实现拆分。

在用户确认该设计文档无误后，下一步应基于本设计生成实现计划，明确：

- 模块拆分
- 技术选型
- 开发顺序
- 测试方案
- 验收标准
